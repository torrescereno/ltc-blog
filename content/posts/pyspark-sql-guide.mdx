Cuando los datos ya no caben en memoria y pandas empieza a matar el proceso, PySpark entra en escena. No es magia: es procesamiento distribuido con una API que, una vez que le agarras el ritmo, se vuelve bastante directa.

Esta guía cubre los patrones que uso en el día a día, con ejemplos sobre un dataset real de Juegos Olímpicos.

<Callout type="info">
  Los ejemplos usan el [dataset de JJ.OO](https://github.com/torrescereno/spark-notes/tree/main/data). Descárgalo para seguir la guía paso a paso.
</Callout>

---

## 1. Crear la sesión

Todo parte aquí. La `SparkSession` es el punto de entrada para leer datos, ejecutar SQL y acceder al contexto de Spark:

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("jjoo-analysis") \
    .getOrCreate()

spark.sparkContext.setLogLevel("ERROR")  # silencia logs innecesarios
```

Un solo `getOrCreate()` por aplicación. Si ya existe una sesión activa, la reutiliza.

---

## 2. Leer datos y definir el schema

### Schema explícito (recomendado)

```python
from pyspark.sql.types import (
    StructType, StructField,
    StringType, TimestampType, IntegerType
)

schema = StructType([
    StructField("date",                TimestampType(), True),
    StructField("stage_code",          StringType(),    True),
    StructField("event_code",          StringType(),    True),
    StructField("event_name",          StringType(),    True),
    StructField("participant_name",    StringType(),    True),
    StructField("participant_country", StringType(),    True),
    StructField("rank",                IntegerType(),   True),
    StructField("result",              StringType(),    True),
])

df = spark.read \
    .format("csv") \
    .option("header", True) \
    .schema(schema) \
    .load("data/results")
```

<Callout type="tip">
  **Por qué definir el schema:** con `inferSchema=True`, Spark lee el archivo completo dos veces para adivinar los tipos. En datasets de millones de filas eso duele. Si ya sabes la estructura, declárala.
</Callout>

### Schema inferido (rápido para exploración)

```python
df = spark.read \
    .option("header", True) \
    .option("inferSchema", True) \
    .csv("data/results")

df.printSchema()  # siempre verifica que Spark infirió bien los tipos
df.show(5)
```

## 3. Consultas SQL

PySpark te deja escribir SQL puro sobre tus DataFrames. Primero registras una vista temporal, después consultas:

```python
df.createOrReplaceTempView("jjoo")

# SQL normal sobre datos distribuidos
result = spark.sql("""
    SELECT
        participant_country,
        COUNT(DISTINCT participant_name) AS athletes
    FROM jjoo
    WHERE rank = 1
    GROUP BY participant_country
    ORDER BY athletes DESC
    LIMIT 10
""")

result.show()
```

Las vistas temporales viven mientras exista la sesión. Para algo persistente entre sesiones, usa `createOrReplaceGlobalTempView` y accede con el prefijo `global_temp.`.

## 4. Filtros y transformaciones

### Filtrar filas

```python
from pyspark.sql import functions as F

# Forma directa
chilenos = df.filter(df["participant_country"] == "Chile")

# Con F.col, más limpio cuando combinas condiciones
finalistas = df.where(
    (F.col("rank") <= 3) &
    (F.col("participant_country").isin(["Chile", "Argentina", "Brasil"]))
)

finalistas.select("participant_name", "event_name", "rank").show(10)
```

`filter` y `where` hacen exactamente lo mismo, es preferencia de estilo. Yo uso `where` cuando las condiciones son compuestas, se lee más natural.

### Modificar columnas

```python
df_clean = df \
    .withColumn("country_upper", F.upper(F.col("participant_country"))) \
    .withColumn("is_medalist", F.col("rank") <= 3) \
    .withColumnRenamed("date", "event_date") \
    .drop("stage_code")   # columna que no necesitamos
```

Encadenar transformaciones así es idiomático en PySpark. Cada método devuelve un nuevo DataFrame, el original no se modifica.

## 5. Agrupaciones y agregaciones

```python
# Disciplinas con más eventos únicos
df.groupBy("discipline_name") \
    .agg(
        F.countDistinct("event_name").alias("unique_events"),
        F.count("participant_name").alias("total_participations")
    ) \
    .orderBy(F.desc("unique_events")) \
    .show(10)
```

```python
# Top países por número de atletas distintos
df.groupBy("participant_country") \
    .agg(F.countDistinct("participant_name").alias("athletes")) \
    .orderBy(F.desc("athletes")) \
    .show(10)
```

`F.countDistinct` evita contar duplicados, que es casi siempre lo que quieres cuando mides participantes.

## 6. Pivot: de filas a columnas

Útil para tablas de resumen donde quieres cruzar dos dimensiones:

```python
# Participantes por género en cada país
# Filas = países, columnas = géneros (M, W, X)
df.groupBy("participant_country") \
    .pivot("gender") \
    .agg(F.count("participant_name")) \
    .orderBy("participant_country") \
    .show(10)
```

El resultado es un DataFrame donde cada valor único de `gender` se convierte en columna. Si conoces los valores posibles de antemano, pásalos explícitamente para que Spark no tenga que calcularlo:

```python
df.groupBy("participant_country") \
    .pivot("gender", ["M", "W", "X"]) \
    .agg(F.count("participant_name")) \
    .show(10)
```

## 7. Optimización de performance

<Callout type="warning">
  **Lazy evaluation:** Spark no ejecuta nada hasta que llamas una acción: `.show()`, `.collect()`, `.count()`, `.write`. Todo lo anterior solo construye el plan de ejecución. Esto es una ventaja, no un bug: Spark puede optimizar el plan completo antes de ejecutarlo.
</Callout>

### Cache: no reprocesar lo que ya calculaste

Si vas a usar el mismo DataFrame varias veces, cachearlo evita recalcularlo desde cero cada vez:

```python
from pyspark import StorageLevel

# Cache básico: memoria + disco si no cabe en RAM
df.cache()

# Control fino sobre dónde almacenar
df.persist(StorageLevel.MEMORY_ONLY)       # solo RAM, más rápido
df.persist(StorageLevel.DISK_ONLY)         # solo disco, más lento pero estable
df.persist(StorageLevel.MEMORY_AND_DISK)   # RAM primero, disco como fallback

# Liberar cuando ya no lo necesitas
df.unpersist()
```

Regla práctica: cachea cuando el DataFrame es resultado de una transformación costosa y lo vas a usar en más de una acción.

### Plan de ejecución

Antes de correr un job pesado, revisa cómo Spark piensa ejecutarlo:

```python
# Ver el plan físico (lo que Spark realmente hará)
df.groupBy("participant_country").count().explain()

# Plan detallado con costos estimados
df.groupBy("participant_country").count().explain(mode="cost")
```

Busca operaciones como `BroadcastHashJoin` (buenas) vs `SortMergeJoin` (más costosas), y `Filter` cerca del inicio del plan, lo que indica que Spark está filtrando antes de procesar.

### Repartition: controlar la distribución

```python
# Redistribuir a 20 particiones balanceadas por columna
df_rep = df.repartition(20, "participant_country")

# Verificar cuántos registros hay por partición
df_rep.foreachPartition(
    lambda rows: print(f"partición con {sum(1 for _ in rows)} registros")
)
```

Más particiones = más paralelismo, pero también más overhead. Como punto de partida, apunta a particiones de entre 100MB y 200MB.

## Patrones que aplico en producción

1. **Schema siempre explícito** en código de producción. `inferSchema` es para notebooks de exploración.
2. **Filtra lo antes posible** en el pipeline. Reducir filas temprano reduce todo lo que viene después.
3. **Cachea con criterio**, no por default. El cache mal usado consume memoria sin beneficio.
4. **Revisa el explain** antes de mandar un job al clúster. Cinco minutos leyendo el plan evitan media hora esperando un job mal construido.
5. **Usa `F.col()` sobre `df["columna"]`** cuando las expresiones son compuestas. Es más legible y evita problemas al referenciar columnas entre DataFrames.

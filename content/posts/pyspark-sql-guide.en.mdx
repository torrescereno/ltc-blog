When data no longer fits in memory and pandas starts killing the process, PySpark steps in. It's not magic: it's distributed processing with an API that, once you get the hang of it, becomes quite straightforward.

This guide covers the patterns I use day to day, with examples built on a real Olympic Games dataset.

<Callout type="info">
  The examples use the [Olympic Games dataset](https://github.com/torrescereno/spark-notes/tree/main/data). Download it to follow the guide step by step.
</Callout>

---

## 1. Creating the session

Everything starts here. The `SparkSession` is the entry point for reading data, running SQL and accessing the Spark context:

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("jjoo-analysis") \
    .getOrCreate()

spark.sparkContext.setLogLevel("ERROR")  # silence unnecessary logs
```

A single `getOrCreate()` per application. If an active session already exists, it reuses it.

## 2. Reading data and defining the schema

### Explicit schema (recommended)

```python
from pyspark.sql.types import (
    StructType, StructField,
    StringType, TimestampType, IntegerType
)

schema = StructType([
    StructField("date",                TimestampType(), True),
    StructField("stage_code",          StringType(),    True),
    StructField("event_code",          StringType(),    True),
    StructField("event_name",          StringType(),    True),
    StructField("participant_name",    StringType(),    True),
    StructField("participant_country", StringType(),    True),
    StructField("rank",                IntegerType(),   True),
    StructField("result",              StringType(),    True),
])

df = spark.read \
    .format("csv") \
    .option("header", True) \
    .schema(schema) \
    .load("data/results")
```

<Callout type="tip">
  **Why define the schema:** with `inferSchema=True`, Spark reads the entire file twice to guess the types. On datasets with millions of rows, that hurts. If you already know the structure, declare it.
</Callout>

### Inferred schema (quick for exploration)

```python
df = spark.read \
    .option("header", True) \
    .option("inferSchema", True) \
    .csv("data/results")

df.printSchema()  # always verify that Spark inferred the types correctly
df.show(5)
```

## 3. SQL queries

PySpark lets you write pure SQL against your DataFrames. First register a temporary view, then query it:

```python
df.createOrReplaceTempView("jjoo")

# Regular SQL on distributed data
result = spark.sql("""
    SELECT
        participant_country,
        COUNT(DISTINCT participant_name) AS athletes
    FROM jjoo
    WHERE rank = 1
    GROUP BY participant_country
    ORDER BY athletes DESC
    LIMIT 10
""")

result.show()
```

Temporary views live as long as the session exists. For something persistent across sessions, use `createOrReplaceGlobalTempView` and access it with the `global_temp.` prefix.

## 4. Filters and transformations

### Filtering rows

```python
from pyspark.sql import functions as F

# Direct form
chileans = df.filter(df["participant_country"] == "Chile")

# With F.col, cleaner when combining conditions
finalists = df.where(
    (F.col("rank") <= 3) &
    (F.col("participant_country").isin(["Chile", "Argentina", "Brazil"]))
)

finalists.select("participant_name", "event_name", "rank").show(10)
```

`filter` and `where` do exactly the same thing — it's a style preference. I use `where` when conditions are compound; it reads more naturally.

### Modifying columns

```python
df_clean = df \
    .withColumn("country_upper", F.upper(F.col("participant_country"))) \
    .withColumn("is_medalist", F.col("rank") <= 3) \
    .withColumnRenamed("date", "event_date") \
    .drop("stage_code")   # column we don't need
```

Chaining transformations like this is idiomatic PySpark. Each method returns a new DataFrame — the original is never modified.

## 5. Grouping and aggregations

```python
# Disciplines with the most unique events
df.groupBy("discipline_name") \
    .agg(
        F.countDistinct("event_name").alias("unique_events"),
        F.count("participant_name").alias("total_participations")
    ) \
    .orderBy(F.desc("unique_events")) \
    .show(10)
```

```python
# Top countries by number of distinct athletes
df.groupBy("participant_country") \
    .agg(F.countDistinct("participant_name").alias("athletes")) \
    .orderBy(F.desc("athletes")) \
    .show(10)
```

`F.countDistinct` avoids counting duplicates, which is almost always what you want when measuring participants.

## 6. Pivot: from rows to columns

Useful for summary tables where you want to cross two dimensions:

```python
# Participants by gender in each country
# Rows = countries, columns = genders (M, W, X)
df.groupBy("participant_country") \
    .pivot("gender") \
    .agg(F.count("participant_name")) \
    .orderBy("participant_country") \
    .show(10)
```

The result is a DataFrame where each unique value of `gender` becomes a column. If you know the possible values in advance, pass them explicitly so Spark doesn't have to compute them:

```python
df.groupBy("participant_country") \
    .pivot("gender", ["M", "W", "X"]) \
    .agg(F.count("participant_name")) \
    .show(10)
```

## 7. Performance optimization

<Callout type="warning">
  **Lazy evaluation:** Spark executes nothing until you call an action: `.show()`, `.collect()`, `.count()`, `.write`. Everything before that only builds the execution plan. This is a feature, not a bug: Spark can optimize the full plan before running it.
</Callout>

### Cache: don't reprocess what you already computed

If you're going to use the same DataFrame multiple times, caching it avoids recomputing from scratch each time:

```python
from pyspark import StorageLevel

# Basic cache: memory + disk if it doesn't fit in RAM
df.cache()

# Fine-grained control over where to store
df.persist(StorageLevel.MEMORY_ONLY)       # RAM only, fastest
df.persist(StorageLevel.DISK_ONLY)         # disk only, slower but stable
df.persist(StorageLevel.MEMORY_AND_DISK)   # RAM first, disk as fallback

# Release when you no longer need it
df.unpersist()
```

Practical rule: cache when the DataFrame is the result of an expensive transformation and you're going to use it in more than one action.

### Execution plan

Before running a heavy job, check how Spark plans to execute it:

```python
# View the physical plan (what Spark will actually do)
df.groupBy("participant_country").count().explain()

# Detailed plan with estimated costs
df.groupBy("participant_country").count().explain(mode="cost")
```

Look for operations like `BroadcastHashJoin` (good) vs `SortMergeJoin` (more expensive), and `Filter` near the start of the plan — which means Spark is filtering before processing.

### Repartition: controlling distribution

```python
# Redistribute to 20 balanced partitions by column
df_rep = df.repartition(20, "participant_country")

# Check how many records are in each partition
df_rep.foreachPartition(
    lambda rows: print(f"partition with {sum(1 for _ in rows)} records")
)
```

More partitions = more parallelism, but also more overhead. As a starting point, aim for partitions between 100MB and 200MB.

## Patterns I apply in production

1. **Always explicit schema** in production code. `inferSchema` is for exploration notebooks.
2. **Filter as early as possible** in the pipeline. Reducing rows early reduces everything that comes after.
3. **Cache with judgment**, not by default. Poorly used cache consumes memory with no benefit.
4. **Check the explain** before sending a job to the cluster. Five minutes reading the plan saves half an hour waiting on a poorly built job.
5. **Use `F.col()` over `df["column"]`** when expressions are compound. It's more readable and avoids issues when referencing columns across DataFrames.
